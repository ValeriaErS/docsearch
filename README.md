# Docsearch— система поиска по документации с использованием RAG

# Назначение проекта 
DocSearch — это утилита командной строки для поиска информации по документации. Она использует подход RAG (Retrieval-Augmented Generation): сначала находит релевантные фрагменты документов по смыслу, а затем формирует ответ с помощью языковой модели, указывая источники. Программа поддерживает локальные эмбеддинги через LM Studio, генерацию ответов через OpenRouter, а также режим mock для тестирования без интернета.

## Веб-интерфейс

DocSearch теперь имеет полноценный веб-интерфейс с авторизацией и чатом.

### Страницы:
- **Главная** — приветственная страница с кнопкой "Начать"
- **Вход** (`/login.html`) — форма входа для существующих пользователей
- **Регистрация** (`/register.html`) — создание нового аккаунта
- **Чат** (`/chat.html`) — диалог с ИИ, история сообщений, источники

### Авторизация и пользователи
- Регистрация и вход через **Supabase** (PostgreSQL в облаке)
- Каждый пользователь видит только **свои документы**
- JWT-токены для сессий
- Инкрементальная индексация — отдельный файл .docsearch_index_{userID}.json для каждого пользователя

## Векторная база Qdrant

Вместо хранения векторов в памяти, DocSearch использует **Qdrant** — векторную базу данных.
Запуск Qdrant: docker run -p 6333:6333 qdrant/qdrant

# Сборка и запуск
## Требования: ##
- Go 1.20 или выше
- LM Studio (опционально, для эмбеддингов)
- API-ключ OpenRouter (опционально, для генерации ответов)
- Docker (для Qdrant) или установленный Qdrant
- Supabase (облачная PostgreSQL)

 ## Установка: ##
- git clone https://github.com/ValeriaErs/docsearch.git
- cd docsearch
- go mod download

 ## Команды: ##
- .\docsearch.exe index --user Имя_пользователя (индексация документов)
- .\docsearch.exe ask "Что такое эмбеддинги?" --user Имя_пользователя (поиск по вопросу)
- .\docsearch.exe ask "Что такое эмбеддинги?" --user Имя_пользователя --out answer.json (сохранение результата в JSON)
- .\docsearch.exe web --port :8080e (запуск HTTP сервера)
- .\docsearch.exe eval --user Имя_пользователя (оценка качества)

## Переменные окружения: ##
- LLM_API_KEY=ваш_ключ
- DATABASE_URL-Строка подключения к Supabase (PostgreSQL)
- JWT_SECRET=ваш_секрет
- Для режима mock установите в конфиге: provider: "mock"

 ## Поля: ##
- query (исходный запрос пользователя)
- answer (ответ LLM с указанием источников)
- sources (использованные фрагменты с документом, оценкой и фрагментом текста)
- model (модель, использованная для генерации)
- tokens_used (количество потраченных токенов)
- duration_ms (время обработки запроса в миллисекундах)

# Краткое описание RAG-пайплайна
Программа загружает документы из папки, убирает из них лишние пробелы и специальные символы, потом режет текст на куски с перекрытием, чтобы не потерять смысл на границах, затем каждый кусок отправляет в LM Studio, чтобы получить эмбеддинг, набор чисел, который описывает смысл текста, и сохраняет всё это в памяти.Когда пользователь задаёт вопрос, программа тоже превращает его в эмбеддинг, потом сравнивает его с эмбеддингами всех кусков с помощью косинусной близости и сортирует результаты так, чтобы самые похожие оказались первыми. Потом программа берёт эти куски, добавляет к каждому номер, чтобы потом можно было ссылаться на источники, собирает их в один большой контекст и передаёт языковой модели. Модель читает контекст и пишет ответ, обязательно указывая, откуда взяла информацию. Если в настройках стоит режим mock, программа не ходит в интернет, а просто возвращает готовый тестовый ответ. Если все найденные куски имеют оценку ниже порога, который задан в конфиге, программа не пытается что-то придумать, а честно говорит, что в документации нет подходящей информации.

# Известные ограничения
1. Qdrant и LM Studio должны быть запущены локально перед работой
2. Поддерживаются только .md, .txt, .pdf, .html — другие форматы не обрабатываются
3. Требуется запущенная LM Studio для эмбеддингов
4. OpenRouter требует интернета, без сети генерация работает только в режиме mock
5. Медленный поиск при большом количестве чанков — поиск выполняется полным перебором, для больших корпусов рекомендуется использовать приближённые индексы

# Безопасность
Конфигурационные параметры хранятся в файле configs/config.yml. Секретные ключи не используются в локальном режиме. При подключении внешних API ключи передаются через переменные окружения и не логируются. Локальный сервер LM Studio работает на локальном адресе и не доступен извне.
